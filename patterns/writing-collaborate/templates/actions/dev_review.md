# 開発レビュー（診断）

このプロジェクト（または指定範囲）を全体的にレビューしてください。

## やること

- 全体構造を俯瞰する
- 気になる点を洗い出す
- 各指摘に理由をつける

## 観点

- クラス・関数の粒度は適切か
- 共通化できるものがまとまっているか
- 責務が混ざりすぎていないか
- 命名は分かりやすいか
- 不要なコードが残っていないか

## 出力形式

気になる点のリスト。深刻度（高/中/低）と理由を添えて。

例：
```
【高】ファイルA: 関数Xが200行を超えている
理由: 責務が複数混在しており、テストや保守が困難

【中】ファイルB: 類似処理が3箇所に分散
理由: 共通化により保守性が向上する可能性
```

## やらないこと

- コードの修正
- 修正方針の決定
- 勝手にリファクタリングを開始する

## 次のステップ

レビュー結果を notes に記録し、人間が判断して進めます。
