# セキュリティチェック

指定範囲（または全体）をセキュリティ観点でチェックしてください。

## やること

- 危険な箇所を洗い出す
- なぜ危険か説明する
- 対処の方向性を示す

## 観点

### シークレット管理
- APIキー、トークン、パスワードがコードや公開場所に漏れていないか
- 環境変数の扱いは適切か
- .env ファイルが .gitignore に含まれているか

### 入力検証
- ユーザー入力をそのまま使っていないか
- バリデーションは適切か
- サニタイズ（無害化）は必要か

### データベース
- SQLインジェクションのリスクはないか
- プリペアドステートメントを使っているか

### 認証・認可
- 認証の抜け穴がないか
- 権限チェックは適切か
- セッション管理は安全か

### その他
- CORS設定は適切か
- XSS（クロスサイトスクリプティング）のリスクはないか
- CSRF（クロスサイトリクエストフォージェリ）対策はあるか

## 出力形式

危険度（高/中/低）と理由、対処の方向性を添えて。

例：
```
【高】ファイルA: 環境変数がハードコードされている
理由: APIキーが公開される可能性
対処: .env に移行し、.gitignore に追加

【中】ファイルB: ユーザー入力を直接HTMLに埋め込み
理由: XSSのリスク
対処: サニタイズ処理を追加
```

## やらないこと

- コードの修正（指摘のみ）
- 過度なセキュリティ要求（現実的な範囲で）

## 次のステップ

チェック結果を notes に記録し、優先度を判断して対処します。
