# 使用例

このディレクトリには、docs-structure パターンの具体的な使用例が含まれています。

## 📄 例ファイル一覧

### [01_example-letter.md](./01_example-letter.md)
申し送りの記入例。Claude Code でのセッション終了時に作成するドキュメントの見本です。

**内容**:
- ユーザー管理機能の開発セッション
- Phase 1（ユーザー一覧ページ）完了
- Phase 2（ユーザー詳細ページ）70%完了
- 次セッションでの検証プロトコル、コマンド実行ルール
- セッション文脈サマリー（設計決定、議論の流れ、空気感）

**ポイント**:
- 前のセッションで何を実装したか具体的に記録
- 次のセッションで何をすべきか明確化
- トークン使用量の管理（75-85%で申し送り検討）
- 「空気感」を言語化して引き継ぐ

---

### [02_example-note.md](./02_example-note.md)
開発ノートの記入例。技術的な問題を解決した際の試行錯誤を記録する見本です。

**内容**:
- Express での REST API 実装パターン
- CORS 設定、エラーハンドリング、ミドルウェアの順序
- 3つのアプローチ（失敗2つ、成功1つ）
- ミドルウェアの正しい配置順序の学び

**ポイント**:
- 失敗した試行錯誤も記録（なぜ失敗したか）
- 最終的な解決策だけでなく、プロセスを残す
- 将来同じ問題に遭遇した時に役立つ情報
- 01_example-letter.md のユーザー管理機能に関連する知見

---

### [03_example-task.md](./03_example-task.md)
タスク管理の記入例。具体的な作業タスクを追跡する見本です。

**内容**:
- API モック実装ファイルの削除タスク
- 背景（本物の API 完成後の後片付け）
- 確認項目・チェックリスト
- 完了条件

**ポイント**:
- 削除前・削除後の確認項目を明記
- 完了条件を具体的に定義
- 関連ドキュメント（申し送り、ノート）へのリンク
- 01_example-letter.md の Phase 1, 2 完了を受けたタスク

---

## 💡 使い方

### 申し送りを作成する時

1. `patterns/docs-structure/templates/letters/TEMPLATE.md` を参考にする
2. [01_example-letter.md](./01_example-letter.md) で具体的な記入例を確認
3. プロジェクトの `docs/letters/YYYY-MM-DD-HH-MM-SS.md` として保存

### 開発ノートを作成する時

1. `patterns/docs-structure/templates/notes/TEMPLATE.md` を参考にする
2. [02_example-note.md](./02_example-note.md) で具体的な記入例を確認
3. プロジェクトの `docs/notes/XX_タイトル.md` として保存（連番を付与）

### タスクを作成する時

1. `patterns/docs-structure/templates/tasks/TEMPLATE.md` を参考にする
2. [03_example-task.md](./03_example-task.md) で具体的な記入例を確認
3. プロジェクトの `docs/tasks/TASK-XXX-タイトル.md` として保存

## 🎯 ポイント

### 申し送りのコツ
- ❌ 曖昧: "だいたい完了"
- ✅ 具体的: "ユーザー一覧ページ実装完了。詳細ページは API 接続検証が残っている"
- 次のセッションの AI が検証できるコマンドを記載
- 「空気感」を言語化（スピード優先？品質優先？）

### 開発ノートのコツ
- 失敗した試行錯誤も記録する（なぜ失敗したか）
- 最終的な解決策だけでなく、プロセスを残す
- 将来同じ問題に遭遇した時に役立つ情報を意識
- 具体的なコード例を含める

### タスクのコツ
- 背景・理由を明確にする（なぜこのタスクが必要か）
- 確認項目をチェックリスト化
- 完了条件を具体的に定義
- 関連ドキュメントへのリンクを記載

## 🔗 3つのドキュメントの関連性

この例では、3つのドキュメントが緩く連携しています：

```
01_example-letter.md（申し送り）
├─ Phase 1, 2 でユーザー管理機能を実装
│
├─> 02_example-note.md（開発ノート）
│   └─ Express での API 実装パターンの知見を記録
│
└─> 03_example-task.md（タスク）
    └─ 本物の API 完成後、モックファイル削除のタスク
```

このように、実際のプロジェクトでもドキュメント間で相互参照することで、文脈を保ちながら情報を整理できます。
