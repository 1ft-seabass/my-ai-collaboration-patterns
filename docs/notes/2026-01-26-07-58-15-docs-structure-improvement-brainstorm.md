# docs-structure 改善の壁打ち

## 現状分析

### 現在の強み
- **AI協働開発に特化した設計**: README駆動でAIが迷わず探索できる
- **実績のある効果**: セッション開始時間80%削減、トークン消費70%削減
- **実用性の高い運用**: このリポジトリで3ヶ月間のドッグフーディング
- **複数の変種**: 用途別に4つのバリエーション（汎用/ブランチ/執筆/アクション専用）

### 使用状況の詳細
- **申し送り**: 6個（2025-10-23 〜 2026-01-21）
- **ノート**: 25個（技術的な試行錯誤の記録）
- **アクション**: 12個（よく使うタスクを自動化）
- **タスク**: 未使用（必要になれば追加予定）

---

## 壁打ちの論点

### 論点1: タスク管理フォルダの活用不足

**観察**:
- `docs/tasks/` が3ヶ月間一度も使われていない
- 代わりに申し送りやノートでタスク管理している可能性

**考えられる原因**:
1. tasks/ の役割が letters/ notes/ と重複している
2. tasks/ を使う必要性が実運用では発生していない
3. セッション単位の作業が多く、長期タスクが少ない

**改善の方向性**:
- Option A: tasks/ を削除し、3フォルダ構成にする（シンプル化）
- Option B: tasks/ の役割を再定義する（プロジェクト管理、マイルストーン等）
- Option C: そのまま（必要になったら使う）

---

### 論点2: ノート管理の課題

**観察**:
- ノートが既に25個あり、増え続けている
- 連番管理（0001〜0025）で時系列はわかるが、テーマ検索が難しい可能性

**考えられる課題**:
1. 「あの技術の試行錯誤、どのノートだっけ?」という検索コスト
2. 関連するノートの横串検索が難しい
3. AIが過去のノートから関連情報を見つけにくい

**改善の方向性**:
- Option A: ノートにタグやカテゴリを追加（frontmatter等）
- Option B: notes/ をサブフォルダでカテゴリ分け（security/, patterns/, refactoring/等）
- Option C: notes/README.md にインデックスを自動生成する仕組み
- Option D: 関連ノートへのクロスリファレンス機能

---

### 論点3: 申し送りの時系列管理

**観察**:
- 申し送りは6個と少なく、タイムスタンプ形式で管理
- アクションの `00_session_end.md` で「最新2件の申し送りを確認」している

**考えられる課題**:
1. 3ヶ月で6個 = 月2回程度のセッション終了
2. 時系列は明確だが、テーマでの振り返りが難しい可能性
3. 将来的に申し送りが100個、200個になったときの管理

**改善の方向性**:
- Option A: letters/ に月別サブフォルダを作成（2025-10/, 2025-11/等）
- Option B: letters/ARCHIVE/ を作成し、古い申し送りを移動
- Option C: letters/INDEX.md で重要な申し送りをキュレーション
- Option D: そのまま（まだ6個なので問題ない）

---

### 論点4: actions の命名規則と発見性

**観察**:
- 連番付き（`00_`, `01_`）+ カテゴリプレフィックス（`git_`, `doc_`）の混在
- ノート0022で「アプローチC: 連番付き + 短縮化」を採用した経緯

**考えられる課題**:
1. よく使う2つだけ連番（00, 01）、他はカテゴリ分け
2. 将来的によく使うアクションが増えたら連番をどう振るか
3. `@actions/` でサジェストされるが、12個すべてを覚えるのは難しい

**改善の方向性**:
- Option A: actions/README.md をユースケース別に整理
- Option B: よく使うランキングTOP5を明示
- Option C: アクションをサブフォルダで分類（session/, git/, doc/, dev/）
- Option D: AIがユーザーの使用頻度を学習して提案（難易度高）

---

### 論点5: パターン間の関係性の複雑化

**観察**:
- docs-structure から4つの派生パターンが生まれている
  - docs-structure（汎用）
  - docs-structure-for-target-branch-only（ブランチ版）
  - writing-collaborate（執筆版）
  - actions-pattern（アクション専用）

**考えられる課題**:
1. 各パターンの使い分けが難しい（初心者にとって）
2. 1つのパターンに改善を入れたとき、他への波及が大変
3. どのパターンを選ぶべきか迷う

**改善の方向性**:
- Option A: パターン選択ガイドを作成（フローチャート形式）
- Option B: 共通部分をモジュール化し、パターン間で共有
- Option C: 統合して1つのパターンに戻す（設定で切り替え）
- Option D: そのまま（用途が明確なので問題ない）

---

### 論点6: セッション文脈サマリーの標準化

**観察**:
- 申し送りに「セッション文脈サマリー」があり、「空気感」まで記録
- これが次セッションの立ち上がりを早くする重要要素

**考えられる改善**:
1. セッション文脈サマリーのフォーマットをより洗練
2. AIが自動で抽出しやすいようにキーワードを構造化
3. プロジェクト固有の情報（技術スタック、制約条件等）を分離

**改善の方向性**:
- Option A: TEMPLATE.md のセッション文脈サマリー部分を強化
- Option B: プロジェクト固有情報を docs/PROJECT.md として別管理
- Option C: letters/ の最新1件を自動で docs/README.md に埋め込む

---

### 論点7: degit による配布と更新の仕組み

**観察**:
- degit でテンプレートを配布する設計
- 一度配布したら、その後の更新は手動

**考えられる課題**:
1. docs-structure に改善を入れても、既存ユーザーには届かない
2. アクションファイルの更新（例: 00_session_end.md の改善）が反映されない
3. バージョン管理の仕組みがない

**改善の方向性**:
- Option A: バージョン番号を導入（v1.0.0等）し、更新通知の仕組み
- Option B: degit ではなく git submodule や npm package で配布
- Option C: actions/ だけは定期的に degit し直すことを推奨
- Option D: そのまま（一度配布したら各ユーザーがカスタマイズする前提）
