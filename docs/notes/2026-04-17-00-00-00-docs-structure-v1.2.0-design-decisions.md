# docs-structure v1.2.0 設計判断の記録

**作成日**: 2026-04-17
**関連タスク**: docs-structure v1.2.0 リリース

## 問題

docs-structure に付随するパターンが増え、同期コストが増大していた。

- `patterns/actions-pattern/` がマスターとして存在するも、docs-structure 側が進化して逆転
- `patterns/docs-structure-for-target-branch-only/` はブランチ用として独立していたが、v1.1.0 の改善を移植しきれていない
- `patterns/writing-collaborate/` は docs/ の衝突回避が目的だったが、その制約がなくなった
- 3パターンの同期維持が現実的でなくなっていた

## 設計判断

### 1. 3パターンの廃止と docs-structure への統合

**判断**: `actions-pattern`・`docs-structure-for-target-branch-only`・`writing-collaborate` を廃止し、docs-structure に統合。

**理由**:
- docs-structure が最も使われており、ここで磨かれた内容が他パターンに追いつけない構造になっていた
- 「マスターからコピー」モデルはメンテコストが高い
- writing-collaborate は docs/ 衝突問題が解消され存在意義がなくなった

**対処**:
- `actions-pattern/WHY.md` の内容は `docs/notes/2025-10-25-00-00-00-actions-pattern-rationale.md` に移植
- `actions-pattern/GUIDE.md` の有用部分は `patterns/docs-structure/GUIDE.md` に吸収

### 2. for_branch_init.md の設計：呼び出しパス統一 + 直接書き換え方式

**問題**: ブランチ作業時に `docs/notes/` ではなく `docs/feature/xxx/notes/` に書きたい。

**検討したアプローチ**:

**アプローチA（採用）**: `for_branch_init.md` が action ファイルを直接書き換える
- docs-structure の action ファイル内の `docs/notes/` → `docs/{branch-name}/notes/` に一括置換
- 呼び出しパスは常に `@docs/actions/` のまま変わらない
- 書き換えは Node.js で行うためクロスプラットフォーム対応

**アプローチB（不採用）**: action ファイルに条件分岐ロジックを追加
- `git branch --show-current` で判定し、`docs/{branch-name}/` が存在すればそちらを使う
- 却下理由: 全 action ファイルに条件分岐が入り複雑化する。main ブランチ利用者にも影響が出る

**アプローチC（不採用）**: action ファイルを `docs/{branch-name}/actions/` にコピー
- 却下理由: マージ後に重複ファイルが残り、@補完に無関係ファイルが混入する

**採用したアプローチA の特徴**:
- ブランチ作業中のみ action ファイルが書き換わる
- マージ前に元に戻すことが望ましいが、事故っても templates から上書き復元できる
- `docs/feature/xxx/` のようにスラッシュはそのままパスになる

### 3. パスを書き換える対象は4ファイルのみ

全 action ファイルを調査した結果、`docs/notes/`・`docs/letters/`・`docs/tasks/` を「書き込み先として参照」するのは以下の4ファイルのみ：

- `00_session_end.md`
- `doc_note.md`
- `doc_letter.md`
- `doc_note_and_commit.md`

`01_git_push.md`（スキャン対象の列挙）や `check_my_security_prepare_level.md`（存在確認程度）は書き換え不要。

## 学び

- 「パターンを独立させてマスター管理」は小規模では機能するが、片方が進化すると同期が破綻する
- 呼び出しインターフェース（パス）を固定して、振る舞いだけ変える設計はシンプルで強い
- 「事故っても復元できる」設計にすることで、cleanup の厳密性を下げられる
