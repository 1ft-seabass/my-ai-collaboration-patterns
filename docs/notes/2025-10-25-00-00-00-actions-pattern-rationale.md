# actions パターンの設計根拠と実測データ

**作成日**: 2025-10-25
**経緯**: patterns/actions-pattern/WHY.md から移植（2026-04-17 patterns/actions-pattern/ 廃止に伴い）

## 核心的な原理

> **「コードで縛る」= 仕様明確 = 議論不要 = トークン節約**

曖昧な会話をやめて明確な指示書に置き換えることで、確認の往復をゼロにする。

### 会話ベース vs actions パターン

| 方式 | フロー | トークン消費 |
|------|--------|-------------|
| 会話ベース | 曖昧な指示 → 確認の往復 → 説明・報告 | 8,000〜10,000 |
| actions パターン | 指示書読み込み → 即実行 → 結果のみ報告 | 2,000〜3,000 |
| **削減率** | | **約70%** |

## 実測データ（joplin-ai-tools、2025年10月）

測定方法: Claude Code のトークンカウンター

| タスク | 会話ベース | actions | 削減率 |
|--------|-----------|---------|--------|
| Git操作 | 6,000 | 1,500 | 75% |
| 知見まとめ | 12,000 | 4,000 | 67% |
| セッション開始 | 8,000 | 2,500 | 69% |
| **平均** | **8,667** | **2,667** | **69%** |

**累計効果（100回実行）**: 600,000トークン節約

## 向いているケース / 向いていないケース

### ✅ actions パターンが有効

- 繰り返し実行するタスク（毎日の Git コミット、セッション開始・終了）
- 手順が決まっているタスク（ビルド → デプロイ → 確認）
- 複数のステップがあるタスク（各ステップに確認が発生するもの）

判断基準: 「このタスクを3回以上実行する予定がある」「手順が明確に定義できる」→ 有効

### ❌ actions パターンが向いていないケース

- 一度きりのタスク（初期セットアップなど）
- 手順が毎回変わるタスク（探索的なバグ調査、新機能の設計）
- 創造的なタスク（ブレインストーミング、初稿執筆）

## なぜ効率的か（4つの理由）

1. **確認の往復ゼロ**: User → AI の 1往復で完結
2. **報告が最小限**: 結果のみ報告（説明・提案なし）
3. **仕様が明確**: 「いい感じに」の解釈不要、毎回同じ品質
4. **コンテキストがクリーン**: 指示書のみ読み込み、過去の会話不要
