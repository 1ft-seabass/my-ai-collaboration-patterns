# setup-securecheck 既存プロジェクトへの再適用で見つかった2つの穴

**作成日**: 2026-07-08
**関連タスク**: setup-securecheck「握りつぶされていた」型バグ対応の追加検証

## 問題

[前回のノート](./2026-07-08-00-00-00-setup-securecheck-suppressed-tool-pattern.md)で `security-verify.js` に `|| true` 検出を追加したが、「README の発動プロンプトを既存プロジェクトにもう一度投げたら、ウィザードが自己修復してくれるか？」を検証したところ、**同じ握りつぶしバグが再発しうる経路が2つ**見つかった。

### 穴1: ステップ3.3 の「マージ」文言が既存の壊れた値を温存する

```
既存の package.json に以下を追加してください：
{ "simple-git-hooks": { "pre-commit": "node scripts/pre-commit.js" }, ... }
注: 既存の scripts とマージしてください（上書きではなく追加）。
```

「追加」「上書きではなく」という指示のため、AI が「`simple-git-hooks` キーはすでに存在する → 設定済みだからスキップ」と判断し、既存の `"... || true"` のような壊れた値を放置してしまうリスクがあった。

### 穴2: Phase 0 判定テーブルが「どのチェックが落ちたか」を見ていない

```
| 10/11 passed（gitleaks のみ ❌）| gitleaks をインストールすれば完了（Phase 1.4 へ） |
```

このテーブルは「10/11 = gitleaks だけが原因」という前提で書かれていた。しかし新設した check #8（`|| true` 検出）が単独で落ちても「10/11 passed」になる。AI が数字だけを見て「gitleaks をインストールすれば完了」と誤判断すると、**Step 3.3（穴1の修正箇所）に一度も到達しないまま完了扱いにしてしまう**。

### 2つの穴の関係

- 穴2（判定テーブル）: Phase 3 まで辿り着く前に「完了」と誤判定される
- 穴1（ステップ3.3）: 仮に Phase 3 まで辿り着いても、既存値を「マージ対象」として温存してしまう

この2つが揃うと、修正した check #8 があっても、既存プロジェクトへの再適用時にバグが再発する経路が残っていた。

## 解決策

### 穴1の修正: ステップ3.3 に既存値の検証・修正指示を追加

```
⚠️ simple-git-hooks キーが既に存在する場合:
値をマージで温存せず、pre-commit の値が node scripts/pre-commit.js と完全一致するか必ず確認してください。
|| true などが付与されている場合、exit code が握りつぶされコミットがブロックされなくなります。
異なっていれば上記の値に修正し、ステップ3.4で再度有効化してください。
```

### 穴2の修正: Phase 0 判定テーブルを「どの項目が❌か」で分岐

```
| 10/11 passed（❌ が gitleaks 関連の項目のみ）| gitleaks をインストールすれば完了（Phase 1.4 へ） |
| 10/11 passed（❌ が gitleaks 関連以外）| 該当項目のみ Phase 1-3 から対応ステップを実施 |
```

数字だけで判断せず、`.git/hooks/pre-commit` の内容チェックが原因の場合は gitleaks と無関係であることを明記。

## 学び

- **「冪等に当て込める」の意味を分解する必要がある**: 「テンプレートファイルの上書きコピー」は素朴に冪等だが、「JSON 設定のマージ」と「判定テーブルの数字ベース分岐」は、既存の壊れた状態を温存・見逃す余地を持ちやすい
- **バグ修正後は「再適用シナリオ」で再検証する**: 検出ロジック（check #8）を直しただけでは不十分で、「その検出結果を受けて実際に直す経路」まで通しで確認しないと、修正が有効化されないまま終わる穴が残る
- **数字ベースの判定は原因の特定を曖昧にする**: 「10/11 passed」のような集計値だけの分岐は、どの項目が原因かによって対応が変わる場面で誤誘導を生む
