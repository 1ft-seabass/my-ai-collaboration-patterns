# 毎コミット自動ネガティブテスト化の構想と実証実験 - 開発記録

> **⚠️ 機密情報保護ルール**
>
> このノートに記載する情報について:
> - API キー・パスワード・トークンは必ずプレースホルダー(`YOUR_API_KEY`等)で記載
> - 実際の機密情報は絶対に含めない
> - .env や設定ファイルの内容をそのまま転記しない

**作成日**: 2026-07-11
**関連タスク**: worktree誤検知修正のコミット後、雑談から発展した構想会話

## 背景

worktree構成でのpre-commit誤検知修正（[[2026-07-11-00-00-00-worktree-precommit-detection-fix]]）を終えた後、「セキュリティチェックの状態がGUIで見えないと人間がチェックしにくい」という話から、CI/CDダッシュボード的なものを作れないかという雑談が始まった。

## 検討の流れ

### 案A: 監視ダッシュボード（GUI/HTML）

最初に出た案は以下の3ページ構成:
1. `index.html` — 導入手順・ネガティブテストのやり方を視覚的にナビゲート
2. `status.html` — 最新の13項目ヘルスチェック結果
3. `history.html` — 直近数コミット分のカバレッジ推移

検討の結果、以下の理由で縮小・撤回した:
- 案1（README的ガイド）は`setup-securecheck.md`と内容が二重管理になる。ちょうど本セッションで「README.mdに古い項目数(12/12)が残っていた」という、まさにドキュメント二重化が原因の不整合を直したばかりだったため、新規に書き起こすのではなく既存mdを見ればよいという結論になった
- 案3（コミットごとの13項目フルカバレッジ）は、`.logs/pre-commit.log`が記録しているのは「pre-commit実行時のpassed/failed/canary」のみで、13項目フルヘルスチェック（`security:verify`）はコミットに紐づかず任意タイミングでの手動実行のため、コミット単位のカバレッジを出そうとするとデータ収集の設計から必要になる。難易度が高いため保留
- 案2（ステータスページ）だけが残ったが、「terminal出力を見るのとあまり変わらない」という指摘があり、単体では着手する価値が薄いと判断

最終的に「**ダッシュボードは、見る人がいて初めて機能する。見なければAIだろうと人間だろうと抜ける**」という結論に至り、ダッシュボード路線自体を却下した。

### 案B: 毎コミット自動でネガティブテストを発動させる

ダッシュボード（観測に依存する仕組み）をやめるなら、「ネガティブテストを毎回必ず自動で回す」方が構造的に確実ではないかという方向に転換した。ただし「他の開発者を巻き込む形で毎コミットに負荷・ノイズが乗る」という開発体験(DX)への懸念が出た。

## 類似サービスの調査

「自分の仕組みと同等以上でサクッと導入できる既存サービスはないか」をWeb検索で確認した。

調査結果（検索結果の質にはSEO記事サイトが多く混じっており玉石混交な点に留意）:
- **TruffleHog**: 検出した値が実際に有効なクレデンシャルかAPIで検証する機能を持つが、これは「漏れた値の生死」の検証であり「検出器自体の生死」の自己検証とは別物
- **GitGuardian**: canarytokens.orgのトークンによる手動テストを推奨事項として案内しているが、コミットごとの自動実行ではない
- **Corgea / Aikido等のAI-nativeAppSecプラットフォーム**: secret検出はSAST等を含む統合スイートの一部で、商用SaaS・比較的重量級

**結論**: 「検出器自体が今も生きてるかを毎コミット自動で証明する」仕組みと「AIとの対話でウィザード式に導入し、既存カスタマイズ済みプロジェクトへも冪等に再適用できる」設計を両方持つ既存サービスは見当たらなかった。

## 実証実験: 毎コミット自動化のコスト測定

### アプローチA: 素朴に別プロセスとしてカナリアスキャンを追加

**試したこと**: `security-verify.js`のcheck#11（機能的カナリアテスト、`gitleaks dir`で合成シークレットを検出）と同じ処理を、`pre-commit.js`内で毎コミット追加実行する

**結果**: 実測でオーバーヘッドが大きいことが判明

**理由**: gitleaksバイナリの起動コスト自体が約0.4〜0.5秒あり、通常の`git --staged`スキャン（実測0.468秒）と、カナリア専用の`gitleaks dir`スキャン（実測0.475秒）がほぼ同じ所要時間。単純に別プロセスとして追加すると毎コミット+0.5秒近く乗ってしまう

---

### アプローチB（成功）: git indexへのカナリアblob注入で単一プロセスに統合

**試したこと**: 使い捨てのscratchリポジトリで、カナリアの内容を**作業ツリーには一切書かず、git indexにのみ**注入し、既存の`gitleaks git --staged`呼び出し1回に混ぜて一緒にスキャンする方式を検証した。

```bash
# カナリアblobをオブジェクトDBに書き込み、indexにのみ登録（作業ツリーは触らない）
BLOB=$(printf 'GITHUB_PAT=ghp_A1b2C3d4E5f6G7h8I9j0K1l2M3n4O5p6Q7r8' | git hash-object -w --stdin)  # gitleaks:allow secretlint-disable-line
git update-index --add --cacheinfo 100644,$BLOB,.canary-probe

# 既存の --staged スキャンをそのまま1回実行するだけで、実ステージ済みファイルと
# カナリアが両方検出される
gitleaks git --staged --config gitleaks.toml --redact .

# 後片付け（作業ツリーには元々存在しないので index から除去するだけでよい）
git update-index --force-remove .canary-probe
```

**結果**: 成功

- カナリアが実ステージ済みファイル（`a.txt`）と一緒に正しく検出された（`leaks found: 1`）
- 所要時間は通常の単独スキャンと同じ約0.46秒（追加のプロセス起動コストがほぼゼロ）
- 除去後、作業ツリー・indexとも注入前と完全に同じ状態に戻ることを確認（`git diff --cached --name-only`で`a.txt`のみに戻っていることを確認）

secretlint側は`git --staged`のようなgit依存モードではなく、単にファイルパスのリストを引数に取る方式のため、カナリア用の一時ファイルパスを既存のスキャン対象リストに1個混ぜ込むだけで同様に実現できる見込み（未検証）。

## 解決策（設計方針、実装は未着手）

「毎コミット、既存の1回のスキャン呼び出しにカナリアを混入させることで、追加のプロセス起動コストなしに検出器の自己検証を行う」という設計が技術的に成立することを確認した。

**実装場所（想定）**: `scripts/pre-commit.js`

## 未解決の注意点

1. **「何らかのリークが見つかった」だけでは自己検証の証拠にならない**: 実ファイルの本物の漏洩とカナリアの検出を区別する必要がある。gitleaksの出力にはパス情報が含まれるため、結果に`.canary-probe`（想定パス名）が含まれているかで判定する必要がある
2. **後片付けの確実性**: プロセスが途中で異常終了した場合でも、indexに注入したカナリアblobが実コミットに紛れ込まないよう、必ずtry/finallyで除去を保証する必要がある。`pre-commit.js`は複数箇所で`process.exit()`を呼ぶ構造のため、既存コードへの組み込み方には注意が必要
3. **パス衝突の回避**: 注入するパス名がユーザーの実ステージ内容と偶然一致しないよう、事前にstaged一覧との衝突有無を確認し、衝突時は自己検証をスキップ（実コミットはブロックしない）する設計が必要

## 学び

- 「毎回自動実行するとコストが乗る」という直感は、素朴な実装（別プロセス追加）では正しいが、既存の呼び出しに混入させる設計に変えるとコストがほぼ消える場合がある。「重いかどうか」を判断する前に、まず実装方式を変えられないか検討する価値がある
- ダッシュボード・監視系の仕組みは「誰かが見る」ことに依存するため、構造的な保証（毎回自動で検証される）に比べて信頼性で劣る、という整理は今後の設計判断の指針になりそう

## 今後の改善案

- `pre-commit.js`へのカナリア混入ロジックの実装（try/finallyでの確実な後片付け、canary-probe専用パスでの検出判定、secretlint側の実装含む）
- 既存の`security-verify.js` check#11（手動実行の機能的カナリアテスト）との役割分担の整理（毎コミット自動版ができた後も、check#11を残す意味があるか要検討）
